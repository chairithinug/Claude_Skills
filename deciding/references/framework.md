# Decision Framework — Step-by-Step Guidance

Detailed guidance for each of the six steps in the /deciding workflow.

---

## Step 1 — Options

List the realistic choices available. Rules:
- Always include "do nothing" if it is genuinely available (it often is).
- Do not invent options the user has not mentioned unless a clearly better
  alternative is being overlooked — name it and ask if they want it included.
- Aim for 2–4 options. More than four dilutes the analysis.

---

## Step 2 — Criteria

Name what a good decision optimizes for, given the user's stated goals and
constraints. Rules:
- Derive criteria from what the user has actually said, not from generic
  decision-making templates.
- Aim for 2–4 criteria. More than four creates false complexity.
- Name criteria as outcomes, not processes (e.g., "low upfront cost" not
  "financial analysis").
- If the user has not stated their goals clearly, infer from context and
  flag the assumption: "Assuming the primary goal is X — correct me if not."

---

## Step 3 — Trade-offs

Evaluate each option against each criterion. Rules:
- Be specific. Avoid false balance — if one option is clearly worse on a
  criterion, say so plainly.
- Use relative language ("Option A is cheaper but slower") not vague language
  ("Option A has financial implications").
- A simple table works when there are 3+ options and 3+ criteria. Otherwise
  prose is clearer and faster to read.

---

## Step 4 — Key Unknown

Name the single most important piece of missing information that, if resolved,
would change the answer. Rules:
- Name one unknown, not a list. Forcing one keeps the framing honest.
- Frame it as a resolvable question: "The key unknown is whether X is true.
  You could resolve this by [doing Y]."
- If there are multiple important unknowns, acknowledge them briefly, then
  still name the one that matters most.

---

## Step 5 — Reversibility Check

Assess which options close off future paths and which preserve optionality. Rules:
- Flag irreversible options explicitly. Irreversibility is not a reason to
  avoid an option — it is a reason to require more certainty before choosing it.
- Options that preserve optionality are more forgiving of being wrong. Note this.
- "Reversible with significant cost" is different from "reversible easily" —
  be specific about what reversal would require.

---

## Step 6 — Pre-mortem

Fast-forward to failure. The decision was made, time has passed, and it turned
out to be wrong. What is the most likely reason? Rules:
- Name one primary failure mode, not a generic risk list.
- Make it specific to this decision — not "execution problems" but "the elder
  users did not trust the vetting process and adoption stalled."
- The pre-mortem is not a reason to avoid the decision. It is a signal for
  what to watch most carefully if the decision is made.
